# Dependency And Compatibility Policy

## Selection Rules

- Prefer latest compatible stable versions.
- Avoid prerelease, nightly, beta, canary, or `0.x` packages unless the project explicitly accepts that risk.
- Check official documentation, release notes, migration guides, compatibility matrices, and known issues.
- Validate with local tests and runtime checks.
- Verify maintenance health, license, security posture, and ecosystem fit for new production dependencies.

## Upgrade Rules

- Patch and minor upgrades still require test evidence.
- Major upgrades require a migration plan and affected-scope analysis.
- Lockfile changes must be generated by the correct package manager.
- Do not mix unrelated dependency upgrades with feature work unless the feature requires them.
- If a dependency introduces broad churn, split the upgrade into its own commit or PR.

## Compatibility Checks

Check compatibility across:

- Runtime version.
- Framework version.
- Language version.
- Build tool and bundler.
- Type system and generated types.
- Peer dependencies.
- Plugin ecosystem.
- Deployment environment.
- Security scanners and policy.

## Rejection Criteria

Reject or ask before using a dependency when:

- It is abandoned or poorly maintained.
- It duplicates a stable project utility.
- It requires broad architecture changes unrelated to the task.
- It hides critical behavior behind magic that is hard to debug.
- Its license or security posture is unclear.
